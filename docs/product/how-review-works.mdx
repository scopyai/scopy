title: "How Scopy works"
description: "The product workflow from GitHub installation to pull request feedback."
---

Scopy runs AI-assisted code review where your team already works: on GitHub pull requests.

## Product flow

1. Connect a GitHub account or organization.
2. Choose the repositories where Scopy should run.
3. Open a pull request or mark a draft pull request ready for review.
4. Scopy analyzes the changed files in enabled repositories.
5. Confirmed findings are posted back to the pull request as review feedback.
6. The product dashboard tracks repositories, pull requests, findings, and analytics for the workspace.

## What Scopy looks for

Scopy is designed to find concrete issues in changed code. The review engine builds context from the pull request diff, affected symbols, repository files, search results, and optional semantic code search before publishing findings.

The strongest findings are specific and actionable:

- They point to the affected file and line.
- They explain the failure mode or risk.
- They focus on behavior introduced or changed by the pull request.
- They avoid broad style commentary unless it affects correctness, security, or maintainability.

## When reviews run

Reviews run automatically for enabled repositories when relevant pull request activity arrives from GitHub, such as a new pull request or a pull request being marked ready for review.

You can also request a fresh review from GitHub by mentioning the GitHub App in a pull request comment.

## Where results appear

Scopy results appear in two places:

- On GitHub, as pull request review comments and a summary comment.
- In the Scopy product, where you can browse repositories, pull requests, review runs, and analytics.

## Workspace model

A workspace represents a connected GitHub account or organization. Repository access, team access, billing, analytics, and review history are scoped to the selected workspace.
